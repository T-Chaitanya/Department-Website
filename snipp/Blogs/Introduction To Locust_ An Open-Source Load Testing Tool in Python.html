Introduction To Locust: An Open-Source Load Testing Tool in Python

A Python-based alternative to Tsung and JMeter for load testing your server

Ng Wai Foong
Jan 3 · 7 min read

Fed up with the existing tools for benchmarking and load testing an existing server, I recently found an open-source tool called Locust to use in one of my projects.

By reading this piece, you’ll be able to use this load testing tool. This provides you with a rough idea of how much requests per second are (RPS) supported by your server. According to the official documentation, Locust is:

“… an easy-to-use, distributed, user load testing tool. It is intended for load-testing web sites (or other systems) and figuring out how many concurrent users a system can handle.

The idea is that during a test, a swarm of locusts will attack your website. The behavior of each locust (or test user if you will) is defined by you and the swarming process is monitored from a web UI in real-time. This will help you battle test and identify bottlenecks in your code before letting real users in.

Locust is completely event-based, and therefore it’s possible to support thousands of concurrent users on a single machine. In contrast to many other event-based apps it doesn’t use callbacks. Instead it uses light-weight processes, through gevent. Each locust swarming your site is actually running inside its own process (or greenlet, to be correct). This allows you to write very expressive scenarios in Python without complicating your code with callbacks.”

There are four sections in this tutorial:

  1. Setup
  2. Basic Usage

Let’s get started!

1. Setup

Installation is pretty straightforward as Locust is supported on Python 2.7, 3.5, 3.6, 3.7, and 3.8. You can simply run the appropriate pip command to install it based on your Python version. It is highly recommended to set up a virtual environment first.

Python 2

python -m pip install locustio

Python 3

python3 -m pip install locustio

Latest master branch from git

python3 -m pip install -e git://github.com/locustio/locust.git@master#egg=locustio

The steps mentioned above should work for Linux and Windows. If things aren’t working as intended, try the following method.

Optional fix for Windows

First, head over to the following website to download the pre-built binary packages for:

  • pyzmq
  • gevent
  • greenlet

You should have the wheel file which can be installed by running the following command for each one of them:

pip install name-of-the-package.whl

Once you are done, simply run the pip install.

pip install locustio

MacOS

MacOS requires gevent to be installed before you can do the pip installation. The easiest way is to install Homebrew:

/usr/bin/ruby -e "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/master/install)"

Then, install libev which is a dependency for gevent:

brew install libev

Then install Locust via pip install.

Test Locust installation

Once you are done, you can test it by running the following code in the command line:

locust --help

Locust Script

Create a new file called locustfile.py. Add the following import statement to it:

from locust import HttpLocust, TaskSet, task, between

Continue writing a class that inherits the TaskSet class. You can name the class based on your preferences.

class TestBehaviour(TaskSet):

Inside the class, you can define your own functions that serve as the task for locust. By default, it provides two base classes that will be called during the start and stop of Locust call:

  • on_start
  • on_stop

If you add the @task decorator on top of a function, it will be treated as a task for Locust. The following example shows a simple task that test the localhost:5000 API:

@task(1)
def hello_world(self):
    self.client.get("http://localhost:5000")

You have to modify the number for the subsequent tasks like the one below:

@task(1)
def hello_world(self):
    self.client.get("http://localhost:5000")@task(2)
def index(self):
    self.client.get("http://localhost:5000/index")

If you would like to set a POST request task, you can use the following code:

self.client.post("/login", {"username":"admin", "password":"admin"})

Once you are done with the it, we need to write another class that inherits HttpLocust. Likewise, you use your desired name for the class. You have to assign two variables:

  • task_set — Set it to the class that inherits from TaskSet. In this case, the TestBehaviour class.
  • wait_time — Refers to the wait time between the execution of task. For now, let’s use between(0.5, 3.0) which indicates a waiting time between 0.5 and 3 seconds. Other built-in functions are available as well, such as constant and constant_pacing. We will change this figure to 0 later for load testing.
class WebsiteTest(HttpLocust):
    task_set = TestBehaviour
    wait_time = between(0.5, 3.0)

Test

Let’s test it by opening another command line and point it to the directory where your locustfile.py is located. If you are using the exact same name, you can simply run:

locust

If you are using a name other than that, you have to use the following command (modify the file name accordingly):

locust -f personal_directory/your_file.py

You should be able to see the following output:

Open up a browser and enter the following URL:

http://localhost:8089/

The following UI will appear. The best practice is to use the actual IP or hostname of the actual host. For now, just fill in anything you like since we are not doing any distributed testing.

Click the Start swarming button once you’re ready. The process will start and you will notice the changes as follows:

  • Request — Total number of requests made so far
  • Fails — Number of requests that have failed
  • Median — Response speed for 50 percentile in ms
  • 90%ile — Response speed for 90 percentile in ms
  • Average — Average response speed in ms
  • Min — Minimum response speed in ms
  • Max — Maximum response speed in ms
  • Average bytes — Average response size in bytes
  • Current RPS — Current requests per second
  • Current Failure/s — Total number of failures per second

Head over to the chart tab and you should see the following charts:

The rest of the menus are for failures and exceptions. You can even download the data in csv via the Download Data tab. You do notice that the RPS is quite low at this moment. This is mainly because we have set the wait_time to be between 0.5 and 3 seconds. For the actual load testing, change both of the values to 0. You can use the constant(0) functions as well based on your preferences. Restart your test and you should be able to see the maximum RPS that your server can handle. Feel free to experiment using a different number of users and hatch rate.

3. Command Line Interface

The module also provides us with a way to run load testing via the command line interface. This allows us to easily implement an automated load testing flow. Simply add the no-web parameters when running locust. You have to also include the host, number of clients, and hatch rate.

locust -f locustfile.py --no-web --host wfng -c 1000 -r 100
  • f — Path to the file
  • no-web — Run the simulation without web interface
  • c — Number of clients
  • r — Hatch rate

Run it for about 30 seconds and stop it using the Ctrl-C command. You should see the following output:

There are other useful parameters available, such as:

  • t — Stop after the specific amount of time. 30s, 5m, 1h, 1h30m.
  • csv — Save the result to files in CSV format.

Feel free to check all the available parameters via:

locust --help

We ran the test and obtained the results in the web UI interface. There are charts and the data can be easily downloaded in CSV format.

Moreover, we tried running the same simulation via just the command line interface. This allows us to run an automated flow for load testing.
